# Безопасность BB V.G.

Репозиторий публичный. В открытых файлах не должны храниться Telegram ID,
chat ID, username пользователей, получатели уведомлений, заявки пользователей,
токены, ключи или содержимое приватного состояния.

## Секреты GitHub Actions

Используются разные секреты для разных задач:

- `BOT_TOKEN` — только доступ к Telegram Bot API;
- `BOT_STATE_KEY` — отдельный ключ шифрования пользовательского состояния;
- `BOT_CHAT_ID` — резервный административный chat ID;
- `ADMIN_USER_ID` — Telegram ID владельца при первичной инициализации.

`BOT_STATE_KEY` не должен совпадать с `BOT_TOKEN`. Рекомендуется случайная строка
не короче 32 байт. Её нельзя отправлять в чат, добавлять в `.env` репозитория или
выводить в GitHub Actions logs.

Старый формат состояния мог использовать `BOT_TOKEN` как ключ. Код умеет читать
его только для безопасной миграции. После добавления `BOT_STATE_KEY` workflow
перезапечатывает данные отдельным ключом. При необходимости временно сохранить
доступ к старому формату перед заменой Telegram-токена используется секрет
`BOT_STATE_LEGACY_KEY`, равный прежнему токену. После успешной миграции этот
временный секрет нужно удалить.

## Шифрование пользовательского состояния

Рабочий формат `bbvg-bot-state-v2` использует:

- AES-256-GCM;
- отдельный случайный 12-байтовый nonce для каждой записи;
- HKDF-SHA256 для получения ключа нужного размера;
- встроенную проверку целостности и подлинности ciphertext.

При неверном ключе, повреждении файла или неподдерживаемом формате бот завершает
загрузку с ошибкой. Повреждённое состояние больше не заменяется незаметно пустым
списком пользователей.

Предыдущий формат `bbvg-bot-state-v1` поддерживается только для чтения и миграции.
Новые записи создаются в формате v2.

## Что хранится

В зашифрованном состоянии могут находиться:

- Telegram ID и chat ID;
- имя и username;
- роль и настройки уведомлений;
- личные отметки участия и скрытые колёса;
- пользовательские заявки на источники;
- история решения по заявке.

В публичных `bot_access.json` и `source_requests.json` персональные разделы должны
оставаться пустыми. Это проверяется в каждом pull request.

## Сроки хранения

- истёкшие личные участия и скрытые колёса удаляются при очередном сохранении;
- персональные поля обработанной заявки на источник обезличиваются через 90 дней;
- ожидающая решения заявка сохраняется до решения или удаления пользователем;
- технический факт решения может оставаться без Telegram ID, chat ID, имени и username.

## Удаление данных пользователем

Обычный пользователь и администратор могут открыть:

`Настройки → Удалить мои данные`

После подтверждения удаляются профиль, роль администратора, настройки,
получатели уведомлений, личные участия, скрытые колёса и ожидающие заявки.
Обработанные заявки остаются только в обезличенном виде.

Владелец не может удалить свои данные, пока не передаст владение другому
зарегистрированному пользователю. Это защищает бот от потери единственного
владельца.

## Автоматические проверки

`security_audit.py` проверяет текущую ветку на:

- Telegram Bot Token-подобные значения;
- приватные ключи;
- отслеживаемые `.env`, session и key-файлы;
- персональные данные в публичных JSON;
- открытые поля внутри контейнера зашифрованного состояния.

Исторический режим выводит только путь и сокращённый hash commit, не содержимое
персональных записей.

Команды:

```bash
python security_audit.py --current
python security_audit.py --history --history-output security_history_report.json
```

## Git-история

В старых commit могли сохраниться предыдущие открытые версии пользовательских
файлов. Текущая ветка обезличена, но это не удаляет данные из истории, старых
клонов, forks и caches.

Переписывание истории — отдельная разрушительная операция. Оно выполняется только
после резервной копии, проверки AES-GCM состояния и отдельного подтверждения
владельца. Порядок описан в `SECURITY_HISTORY_CLEANUP_RU.md`.

## Аварийные действия

Если утёк `BOT_TOKEN`:

1. немедленно отозвать его через `@BotFather`;
2. выпустить новый токен;
3. заменить GitHub Secret `BOT_TOKEN`;
4. проверить, что состояние уже использует `key_mode=dedicated`;
5. перезапустить панель и монитор.

Если утёк `BOT_STATE_KEY`:

1. остановить запись состояния;
2. сохранить зашифрованную резервную копию;
3. расшифровать текущим ключом в контролируемой среде;
4. создать новый случайный ключ;
5. повторно зашифровать состояние;
6. заменить secret и проверить восстановление.

Никогда не добавляйте в репозиторий `.env`, Telegram session-файлы, приватные
ключи, локальную виртуальную среду или незашифрованные экспорты пользователей.
