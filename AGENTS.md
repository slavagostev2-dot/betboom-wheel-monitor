# BB V.G. — обязательные инструкции для работы с репозиторием

Этот файл нужно прочитать **до любых изменений**. Он задаёт архитектурные и эксплуатационные правила проекта.

## 1. Главный принцип

Не создавайте новые файлы с версиями, временными названиями или словами `final`, `new`, `fix`, `copy`, если для функции уже существует подходящий модуль.

Запрещённые примеры: `admin_panel_runtime_v42.py`, `monitor_new.py`, `final_fix.py`, `chapter6.py`, `test2.py`.

По умолчанию изменяйте существующий тематический модуль. Новый файл допустим только если одновременно выполнены условия:

- существующий модуль не соответствует новой ответственности;
- добавление кода в существующий файл ухудшит связность, тестируемость или безопасность;
- новому модулю можно дать устойчивое предметное название без номера версии;
- определена папка, владеющая этой ответственностью;
- обновлены архитектурная карта и журнал изменений.

Перед созданием нового файла письменно объясните в итоговом отчёте, почему нельзя расширить существующий модуль. Цель проекта — уменьшать или сохранять число файлов.

tests/

## 3. Действующие точки входа до завершения рефакторинга

- Telegram-панель: `bbvg/bot/runtime.py` через временные совместимые runtime-слои; production-совместимая команда пока остаётся `admin_panel_runtime_v41.py`.
- Монитор колёс: `bbvg_monitor_main.py`, `monitor.py` и тематические модули.

Не добавляйте параллельный runtime или второй consumer Telegram `getUpdates`.

## 4. Правила изменений

Перед изменением:

1. Найдите все импорты и ссылки на файл.
2. Проверьте связанный workflow.
3. Определите затрагиваемые runtime-данные.
4. Сохраните callback-данные и формат состояния.
5. Проверьте, можно ли изменить существующий модуль без нового файла.
6. Для крупного обновления создайте отдельную резервную ветку и зафиксируйте её в разделе «Актуальные бэкапы» этого файла.

После изменения:

1. Скомпилируйте изменённые Python-файлы.
2. Запустите относящиеся unit/integration/scenario tests.
3. Запустите полный `pytest` и production acceptance.
4. Проверьте панель и свежий heartbeat.
5. Проверьте монитор, число источников и свежий heartbeat.
6. Обновите `docs/PROJECT_CHANGELOG_RU.md` при изменении поведения, архитектуры, структуры, workflow, состояния или эксплуатации.
7. Обновите `AGENTS.md` и документы в `docs/`, если изменились правила, точки входа, зависимости, расположение модулей или актуальная точка отката.

Развёртывание нельзя считать успешным только по коммиту или компиляции: нужен подтверждённый живой процесс.

## 5. Обязательные бэкапы крупных обновлений

Перед каждым крупным обновлением, рефакторингом, миграцией состояния, изменением workflow или массовым удалением файлов обязательно:

1. Создать отдельную ветку вида `backup/YYYY-MM-DD-description` или `backup/before-description`.
2. Указать точный commit SHA, дату, назначение и последнее подтверждённое тестовое состояние.
3. Не перемещать backup-ветку после начала крупного обновления.
4. После успешного обновления создать новую актуальную backup-точку уже от подтверждённого результата либо явно оставить прежнюю, если production ещё не переключён.
5. Обновить этот раздел и запись в `docs/PROJECT_CHANGELOG_RU.md`.

Откат крупного изменения выполняется возвратом ветки/коммита целиком. Нельзя восстанавливать набор файлов вручную по памяти, если существует зафиксированный backup.

## 6. Обязательный журнал проекта

`docs/PROJECT_CHANGELOG_RU.md` — единый журнал значимых изменений. Новый чат обязан прочитать его сразу после `AGENTS.md`.

В журнал вносятся:

- новые пользовательские возможности;
- изменения обнаружения, дедупликации, таймеров и рейтинга;
- исправления поведения;
- создание, перенос, объединение, переименование и удаление модулей;
- изменения точек входа, зависимостей, workflow, хранения и миграций;
- создание новой backup-точки, её SHA, назначение и заменяемый предыдущий backup;
- важные откаты и отклонённые решения.

Не вносятся heartbeat-коммиты, обычные сохранения runtime JSON и чистое форматирование.

Запись содержит дату, название, причину, изменения, затронутые модули, проверки, актуальный backup и способ отката. Новые записи добавляются сверху. Значимая работа без записи в журнале не считается завершённой.

## 7. Обязательные контракты колёс

- одинаковый Telegram-пост повторно не отправляется;
- одинаковая ссылка с автоматическим таймером блокируется до прокрутки;
- ручной таймер имеет приоритет;
- без таймера одинаковая ссылка блокируется на 2 часа;
- после окончания окна ссылка может обозначать новое событие;
- новое событие не наследует участие и публикации старого.

## 8. Telegram-интерфейс

- Не изменяйте `callback_data` без миграции и тестов.
- Лимит callback — 64 байта.
- Не используйте декоративные цвета для связи колеса с кнопками.
- Значок должен обозначать состояние или действие.
- Повторное административное действие не начисляет рейтинг повторно.

## 9. Состояние и безопасность

- Не публикуйте токены, chat ID, user ID и расшифрованное состояние.
- Не заменяйте `BOT_STATE_KEY` токеном бота.
- Не удаляйте runtime JSON без миграционного анализа.
- Формат состояния должен быть обратимо мигрируемым.

## 10. Разрешённые эксплуатационные действия

Разрешено при необходимости временно остановить или перезапустить панель/монитор, отправить ограниченное тестовое уведомление, выполнить тестовое административное действие и включить временную диагностику.

Условия: минимальное воздействие, метка `ТЕСТ`, отсутствие массовой рассылки без отдельного запроса, удаление временной диагностики, подтверждение heartbeat и наличие точки отката.

## 12. Удаление файлов

Файл удаляется только если он не импортируется, не указан в workflow, не требуется проверками, полезное содержимое перенесено, полный прогон проходит, а процессы оставляют свежий heartbeat.

Git хранит историю, поэтому старые версии не нужно сохранять в основной ветке ради архива.
