# BB V.G. — обязательные инструкции для работы с репозиторием

Этот файл нужно прочитать **до любых изменений**. Он задаёт архитектурные и эксплуатационные правила проекта.

## 1. Главный принцип

Не создавайте новые файлы с версиями, временными названиями или словами `final`, `new`, `fix`, `copy`, если для функции уже существует подходящий модуль.

Запрещённые примеры:

- `admin_panel_runtime_v42.py`
- `monitor_new.py`
- `final_fix.py`
- `chapter6.py`
- `test2.py`

По умолчанию изменяйте существующий тематический модуль. Новый файл допустим только если одновременно выполнены условия:

- существующий модуль не соответствует новой ответственности;
- добавление кода в существующий файл ухудшит связность, тестируемость или безопасность;
- новому модулю можно дать устойчивое предметное название без номера версии;
- определена папка, владеющая этой ответственностью;
- обновлены архитектурная карта и журнал изменений.

Перед созданием нового файла письменно ответьте в отчёте: почему нельзя расширить существующий модуль. Цель проекта — уменьшать или сохранять число файлов, а не увеличивать его без необходимости.

## 3. Где находится логика

До завершения рефакторинга действующая система состоит из двух процессов:

- Telegram-панель: `admin_panel_runtime_v41.py` и её текущая цепочка наследования;
- монитор колёс: `bbvg_monitor_main.py`, `monitor.py` и подключаемые тематические модули.

Не добавляйте ещё один параллельный runtime или второй consumer Telegram `getUpdates`.

## 4. Правила изменений

Перед изменением:

1. Найдите все импорты и ссылки на изменяемый файл.
2. Проверьте соответствующий workflow в `.github/workflows/`.
3. Определите, какие runtime-файлы состояния могут измениться.
4. Сохраните совместимость callback-данных Telegram и формата состояния.
5. Проверьте, можно ли внести изменение в уже существующий тематический модуль без создания нового файла.
6. Перед крупным обновлением создайте отдельную backup-ветку и внесите её в раздел «Актуальные бэкапы» этого файла.

После изменения обязательно:

1. Запустить компиляцию изменённых Python-файлов.
2. Запустить unit/integration/scenario tests, относящиеся к изменению.
3. Запустить полный pytest и production acceptance.
4. Проверить запуск Telegram-панели.
5. Проверить запуск монитора и свежий heartbeat.
6. Убедиться, что проверяются все настроенные источники и нет новых ошибок.
7. Обновить `docs/PROJECT_CHANGELOG_RU.md`, если изменение затрагивает поведение, архитектуру, структуру, workflow, состояние или эксплуатацию.
8. Обновить `AGENTS.md` и документацию, если изменились правила, точки входа, зависимости, расположение модулей или актуальная точка отката.

Нельзя считать развёртывание успешным только по коммиту или успешной компиляции. Нужен подтверждённый живой процесс.

## 5. Обязательные бэкапы крупных обновлений

Перед каждым крупным обновлением, рефакторингом, миграцией состояния, изменением workflow или массовым удалением файлов обязательно:

1. Создать отдельную ветку вида `backup/YYYY-MM-DD-description` или `backup/before-description`.
2. Зафиксировать точный commit SHA, дату, назначение и последнее подтверждённое тестовое состояние.
3. Не перемещать backup-ветку после начала крупного обновления.
4. После успешного обновления создать новую backup-точку от подтверждённого результата либо явно оставить прежнюю, если production ещё не переключён.
5. Обновить этот раздел и `docs/PROJECT_CHANGELOG_RU.md`.

Откат крупного изменения выполняется возвратом ветки или коммита целиком. Нельзя восстанавливать набор файлов вручную по памяти, если существует зафиксированный backup.

## 6. Обязательный журнал проекта

`docs/PROJECT_CHANGELOG_RU.md` — единый понятный журнал значимых изменений проекта. Новый чат обязан прочитать его после `AGENTS.md`.

В журнал вносятся:

- новые пользовательские возможности;
- изменения правил обнаружения, дедупликации, таймеров и рейтинга;
- исправления, влияющие на поведение системы;
- создание, перенос, объединение, переименование и удаление модулей;
- изменения точек входа и зависимостей;
- изменения GitHub Actions, хранения, миграций и эксплуатационных процедур;
- создание новой backup-точки, её SHA, назначение и заменяемый предыдущий backup;
- важные откаты и причины отказа от решений.

Не вносятся:

- автоматические heartbeat-коммиты;
- обычные сохранения runtime JSON;
- форматирование без изменения смысла;
- временная диагностика, полностью удалённая в той же работе, кроме случаев, когда её результат важен для архитектуры.

Каждая запись должна содержать дату, краткое название, причину, фактические изменения, затронутые файлы или модули, проверки, актуальный backup и способ отката. Новые записи добавляются сверху.

Работа не считается завершённой, если значимое изменение не отражено в этом журнале.

## 7. Правила дедупликации колёс

Текущий обязательный контракт:

- одинаковый Telegram-пост повторно не отправляется;
- одинаковая ссылка при найденном автоматическом таймере блокируется до прокрутки;
- ручной таймер администратора имеет приоритет;
- без таймера одинаковая ссылка блокируется на 2 часа;
- после окончания окна та же ссылка может обозначать новое колесо;
- новое событие не должно наследовать участие и публикации старого события.

Эти правила нельзя ослаблять без отдельного согласования.

## 8. Telegram-интерфейс

- Не изменяйте `callback_data` без миграции и тестов.
- Максимум Telegram для callback — 64 байта.
- Не используйте декоративные цвета для связи колеса с кнопками; связь строится по номеру.
- Цвет и значок должны обозначать только состояние или действие.
- Повторное административное действие не должно повторно начислять рейтинг.

## 9. Состояние и безопасность

- Не публикуйте токены, chat ID, user ID и расшифрованное приватное состояние.
- Не заменяйте отдельный `BOT_STATE_KEY` токеном бота.
- Не удаляйте runtime JSON без анализа миграции.
- Изменения формата состояния должны быть обратимо мигрируемыми.

## 10. Разрешённые эксплуатационные действия

Владелец проекта разрешил при необходимости:

- временно остановить Telegram-панель или монитор;
- перезапустить GitHub Actions workflow;
- отправить ограниченное тестовое уведомление в бот;
- выполнить тестовые административные действия;
- временно включить диагностический workflow.

Условия:

- использовать минимально необходимое воздействие;
- явно помечать тестовые уведомления словом `ТЕСТ`;
- не отправлять массовые уведомления пользователям без отдельного запроса;
- после теста удалить временные workflow и диагностические файлы;
- подтвердить восстановление свежим heartbeat;
- перед рискованной операцией иметь точку отката.

## 12. Удаление файлов

Файл можно удалить только если одновременно выполнены все условия:

- он не импортируется Python-кодом;
- он не указан в workflow;
- он не требуется preflight/acceptance-тестами;
- полезная логика и документация перенесены;
- полный тестовый прогон проходит без него;
- рабочие процессы запускаются и оставляют свежий heartbeat.

История Git сохраняет удалённые файлы, поэтому старые версии не нужно хранить в основной ветке только ради архива.
