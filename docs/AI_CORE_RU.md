# BB V.G. — AI Core

## Назначение

`bbvg/ai_core.py` — единый необязательный шлюз для будущих AI-функций проекта.
Он не владеет обнаружением колёс, Telegram-уведомлениями, автоучастием или
жизненным циклом колёс и не должен становиться обязательной зависимостью этих
контуров.

Этап 0 создаёт только фундамент. Ни одна пользовательская AI-функция пока не
включена.

## Главный контракт безопасности

Если AI выключен, не настроен, превысил лимит, вернул ошибку или некорректный
JSON, вызывающий код обязан получить детерминированный fallback. Исключение из
AI Core не должно останавливать production-монитор.

По умолчанию `BBVG_AI_ENABLED=false`, а список включённых AI-функций пуст.

## Зарезервированные feature flags

- `health_inspector` — будущий AI-инспектор здоровья системы;
- `suspicious_post_analysis` — будущий анализ сомнительных публикаций;
- `source_discovery` — будущий поиск и оценка новых источников;
- `natural_language_admin` — будущий разбор администраторских команд на естественном языке.

Глобальное включение: `BBVG_AI_ENABLED=true`.

Список функций: `BBVG_AI_FEATURES=health_inspector,suspicious_post_analysis`.
Также поддерживаются индивидуальные переменные вида
`BBVG_AI_FEATURE_HEALTH_INSPECTOR=true`.

Feature flag сам по себе не должен давать AI право изменять production-state.
Каждая будущая функция отдельно определяет read-only или подтверждаемый
исполнительный контракт.

## Провайдер

Начальный провайдер — OpenAI Responses API через HTTPS. Ключ читается только из
`OPENAI_API_KEY`, модель — из `BBVG_AI_MODEL`. Ключ никогда не записывается в
runtime-state, журнал решений или логи AI Core.

Если ключ или модель не заданы, AI Core возвращает `not_configured` и fallback.

## Кэш

Кэш предназначен для одинаковых AI-запросов и ограничивает лишние обращения к
провайдеру. Ключ кэша — SHA-256 от feature, провайдера, модели и полного запроса.
Сам входной prompt в журнал решений не записывается.

По умолчанию TTL кэша — 900 секунд. Настройка:
`BBVG_AI_CACHE_TTL_SECONDS`.

## Лимит запросов

AI Core содержит общий минутный лимит. По умолчанию — 10 новых запросов в
минуту. Попадание в кэш лимит не расходует.

Настройка: `BBVG_AI_MAX_CALLS_PER_MINUTE`.

При превышении лимита возвращается `rate_limited` и fallback.

## Журнал решений

Состояние хранится в `ai_runtime_state.json` и ограничивается последними
записями. По умолчанию сохраняется не более 500 решений.

Журнал хранит:

- ID решения;
- время;
- feature;
- статус;
- провайдера и модель;
- факт использования кэша;
- SHA-256 результата;
- безопасные структурированные поля `decision`, `classification`, `action`,
  `reason`, `confidence`, если они присутствуют в JSON-ответе;
- краткую ошибку провайдера при сбое.

Исходный prompt намеренно не сохраняется в decision log.

Настройка размера журнала: `BBVG_AI_MAX_DECISIONS`.

## Structured JSON

`AIClient.ask_json()` добавляет требование вернуть один JSON-объект. Если ответ
не разбирается как объект, вызывающий код получает `invalid_json` и переданный
fallback-объект.

AI-ответ не считается доверенным только потому, что он валидный JSON. Будущие
модули обязаны валидировать допустимые enum, диапазон confidence и разрешённые
поля на своей предметной границе.

## Runtime state

Начальная схема `ai_runtime_state.json`:

```json
{
  "cache": {},
  "decisions": [],
  "rate_calls": [],
  "version": 1
}
```

Файл не содержит API-ключей. До подключения первой реальной AI-функции он не
включён в обязательный production runtime push: этап 0 не должен менять работу
текущих long-running процессов. Интеграция persistence с конкретным workflow
выполняется одновременно с первой использующей AI функцией.

## Тестовый контракт

`tests/test_ai_core.py` проверяет:

- отключённый AI не вызывает провайдера и не создаёт state-файл;
- повтор одинакового запроса обслуживается из кэша;
- минутный rate limit возвращает fallback;
- JSON-ответ разбирается, а classification/confidence попадают в журнал;
- ошибка провайдера не пробрасывается в вызывающий production-код.
